## Key Implementation Details

### 1. **Bill Parsing** (`app/bill_parser.py`)

Uses Claude's structured outputs (JSON schema) for guaranteed valid extraction:

```python
# Calls Claude vision API with schema validation
response = client.messages.create(
    model="claude-opus-4-8",
    output_config={"format": {"type": "json_schema", "schema": BILL_SCHEMA}},
    messages=[{
        "role": "user",
        "content": [
            {"type": "image", "source": {...}},  # Base64 JPEG
            {"type": "text", "text": "Extract items..."},
        ],
    }],
)
```

**Features:**
- Resizes images (max 1568px longest edge) to reduce token cost
- Validates schema at API level (not client-side JSON parsing)
- Handles errors gracefully (refusal, max_tokens, connection)
- Converts to `Decimal` for precise money arithmetic

### 2. **Cost Calculation** (`app/calculations.py`)

Implements proportional tax/tip split based on claimed items:

1. Each claimed item is split evenly among claimers
2. Each participant's claimed subtotal = sum of their claimed shares
3. Tax/tip ratio = participant's claimed / total claimed (handles unclaimed items)
4. Total = claimed + tax_share + tip_share

**Example (Burger $10 + Fries $5 + Coke $3):**
- A claims Burger + Coke ($13) → ratio 13/18
  - Tax: $1.80 × 13/18 = $1.30
  - Tip: $3.60 × 13/18 = $2.60
  - **Total: $16.90**
- B claims Fries ($5) → ratio 5/18
  - Tax: $1.80 × 5/18 = $0.50
  - Tip: $3.60 × 5/18 = $1.00
  - **Total: $6.50**
- **Sum: $23.40** ✓

### 3. **Session Storage** (`app/store.py`)

In-memory dictionary with `threading.Lock` for thread safety:

```python
SESSIONS: dict[str, Session] = {}
_lock = threading.Lock()
```

**Design choices:**
- Single process only (no multi-worker deployment)
- Sessions lost on restart (acceptable for a single-dinner prototype)
- Lock ensures consistent reads/writes across threads
- No database complexity

### 4. **Live Updates via Polling** (`static/app.js`)

Vanilla JavaScript fetches updated HTML every 2 seconds:

```javascript
async function fetchFragment() {
  const response = await fetch(`/sessions/${sessionId}/fragment`);
  const html = await response.text();
  document.getElementById("items-container").innerHTML = html;
}
setInterval(fetchFragment, 2000);
```

**Why polling instead of WebSockets?**
- Simpler — no connection lifecycle/reconnect logic
- Adequate latency for social interaction (2–3 seconds acceptable)
- Same single-process constraint already present (in-memory store)
- Immediate feedback on user's own claim (fetch after submit)

### 5. **Frontend** (Jinja2 + vanilla JS/CSS)

- **Server-side rendering** — all HTML generated by Jinja2 (no API-only)
- **Mobile-first** — responsive CSS, touch-friendly buttons
- **Camera capture** — `<input type="file" capture="environment">` opens phone camera
- **QR code** — generated server-side as PNG, embedded in template
- **No build step** — plain HTML/CSS/JS, works immediately

### 6. **API Routes** (`app/routes/api.py`)

## Deployment Considerations

### ✅ What Works Now

- Single-dinner use case (one session per run)
- LAN-only (HTTP works, no HTTPS needed for local dev)
- Mobile and desktop browsers
- Image uploads from phone camera or file system
- Real-time updates (polling acceptable for social scenario)

### ⚠️ Before Production

If you want to deploy this publicly:

1. **Database** — replace in-memory store with PostgreSQL/SQLite
2. **HTTPS** — use mkcert locally or real cert for production
3. **Multi-worker** — add database session locking if using multiple processes
4. **Error handling** — add logging, sentry, structured errors
5. **Rate limiting** — throttle bill uploads (Claude API costs)
6. **Authentication** — add user accounts if needed
7. **Persistence** — store receipts, history, user sessions

### Single-Process Requirement

The app uses `threading.Lock` on an in-memory dict. For deployment:

```bash
# CORRECT: single process
uvicorn app.main:app --host 0.0.0.0 --port 8000

# WRONG: multiple workers (inconsistent state)
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```

If you add a database, you can use `--workers N > 1`.

## Known Limitations

1. **Whole items only** — can't split "1 of 3 fries" (per-unit splitting not implemented)
2. **No accounts** — anyone with the link can join
3. **No persistence** — sessions lost on restart
4. **LAN only** — designed for local use, not public internet
5. **Single model** — hardcoded to `claude-opus-4-8` (configurable in `config.py`)
6. **No async** — bill parsing blocks the request (acceptable for one-time call)

## Development Notes

### Why These Choices?

1. **FastAPI** — modern, fast, built-in automatic API docs
2. **Jinja2** — server-side templates (simpler than JS frameworks for this scale)
3. **Claude vision API** — most accurate for money-sensitive OCR
4. **Structured outputs** — guaranteed JSON validation at API level
5. **Polling** — simpler than WebSockets for this use case
6. **Decimal** — precise money arithmetic (no float rounding errors)
7. **threading.Lock** — simple concurrency for in-memory store
8. **Vanilla JS** — no build step, works immediately on any browser

### Why NOT These Choices?

- **Streamlit** — not suitable for real-time multi-user interactions
- **React/Vue** — overkill for this single-page scenario
- **WebSockets** — same single-process constraint, added complexity
- **Tesseract/Google Vision** — Claude more accurate for structured data
- **SQLite** — adds complexity for a prototype
- **Kubernetes** — not needed for single-process, LAN-only app
